Release Notes

Note: prior release notes are appended in reverse order below these notes.

The is the fourth Alpha release of Sets in Motion.

Features Planned

  1. More calls.  Advanced-2 is up next.

Features Added

  1. The A-1 list has been added to the definitions.
  2. Call definitions can use two forms of action now: the original single expression for simple actions, or a new multi-track action for definitions where groups of two or four dancers at a time are performing distinct moves.
  3. You can drag tabs from one side of the window to the other to customize the layout as you like.
  4. The program will now do calculations of the call results without locking up the user interface.  This makes startup much faster.
  5. Dance files (such as your Library) will store the last checked status.  That way, if you don't modify the definitions between runs of the program, it doesn't have to recheck every call every time the program starts.

 

In the following lists, the letters SD1, SD2, SD3 denote bugs in square dance choreography while UI1, UI2, UI3 denote user interface bugs.  The number indicates the severity of the bug.  Likelihood of encountering the bug is not considered in severity.  Severity describes how badly the user is impacted when the bug does occur.  The impact criteria for severity settings are intended to be objective, but borderline cases will always present themselves. 

Severity should be interpreted according to the following scale:

  1. Severity 1 bugs are catastrophic.  The user can lose data entirely including unsaved edits and/or file contents.  Data may not be obliterated completely but may be inaccessible to the user.  So, a bug that leaves on-disk data intact, but causes the program to crash on startup every time would have severity 1.
  2. Severity 2 bugs are blocking functionality.  Data that you have entered is still present and accessible, but some discrete function does not work correctly and no work around is possible to achieve the desired function.  A display field that shows incorrect data, for example, would be blocking if the information was not available anywhere else in the program.
  3. Severity 3 bugs are non-blocking.  Data you have entered is present and accessible.  Some discrete function does not work or some display is misleading, confusing or malformed.  However, any obscured information is available by some other means and actions you want taken can be done somehow using the program.

Known Bugs

  1. SD2 Four-dancer stars are not supported in most circumstances..
  2. SD2 Couples are not defined for infacing rings, where boy-girl arrangements should be able to do couple-based calls.
  3. SD2 From a squared set, the call 'turn your partner by the left 1/2' does not produce a thar.  It should.
  4. SD2 Dancers move through one another without collision.  This is also impossible.
  5. SD2 Spread does not correctly take into account how the previous call ended.  It should.
  6. SD2 Do your part does not work correctly if a counter-part dancer cannot be found when the facing of a phantom is uncertain.
  7. UI2 Scroll bars in the Definition Editor get misaligned.  Some parts of large, long definitions are inaccessible.
  8. UI3 The headings in the Dance Editor are misaligned.
  9. UI3 The Formation Editor will sometimes shift the displayed position of a formation if new spots are added (this sometimes shifts the spots completely out of view).  Saving and restarting the program fixes the problem.
  10. UI3 Animations of a weave the ring do not show correct motions (but do not fail, so the problem is in the partial results).
  11. UI3 The animation for six-two acy deucey does not work correctly for four of the dancers.  The final position is, however, correct.

Bugs Fixed

  1. SD2 U-turn back does not properly account for dancer flow.
  2. UI3 Scroll bars in the Grammar Editor grow much faster than the size of the window shrinks.  Scroll bars get created precisely when they are needed, but as the window shrinks further, the scroll bar overcompensates and a large white space develops as you scroll to the bottom.
  3. UI2 The text displayed for an ampersand is not correct (essentially the ampersands disappear).
  4. UI3 The tab heading for a new sequence does not update when you change the comment.

 

 

Alpha-3 Release Notes- July 3, 2010

The is the third Alpha release of Sets in Motion.

Features Planned

  1. More calls.  Advanced is up next.

Features Added

  1. The Plus list has been added to the definitions.
  2. The exact version of the product is now displayed in the lower right corner of the main window of the program.
  3. A Dancer Designator editor has been added.  Now when you expand the Dancer Designations section of a grammar and click on a dancer designator, a Designator editor will be brought up in the left-hand sub-window. 
  4. A variety of dancer designations have been added that allow you to more precisely describe which dancers you want to perform an action.  Combinations such as lead center heads can be typed.  You can also specify two or more groups, as in boys and side girls.
  5. It is now possible to write formations with Inactive dancers in them (see the center_box formation and the revised definition for Box Circulate for an example of how this is used).
  6. A Performance Window has been added.  When you select one or more sequences you can choose to Perform them, which will open a separate window where you can read the calls in sequence with a minimum of additional detial to get in the way. 
  7. Preferences have been added to allow you to set the fonts used in the Performance Window.
  8. When you create a new Sequence, the editor now automatically puts you on the first line of the sequence, shows the animation and sets the level to the level of the last sequence you had in the Sequence Editor.  This should allow you to quickly get started on a new sequence.
  9. The sequence editor has been enhanced with a new area below the calls where a list of possible completions of your current typing are shown.  These lists update as you change the sequence level or type in more text.

 

Known Bugs

  1. SD2 Four-dancer stars are not supported in most circumstances..
  2. SD2 Couples are not defined for infacing rings, where boy-girl arrangements should be able to do couple-based calls.
  3. SD2 From a squared set, the call 'turn your partner by the left 1/2' does not produce a thar.  It should.
  4. SD2 Dancers move through one another without collision.  This is also impossible.
  5. SD2 Spread does not correctly take into account how the previous call ended.  It should.
  6. SD2 U-turn back does not properly account for dancer flow.
  7. UI3 Scroll bars in the Grammar Editor grow much faster than the size of the window shrinks.  Scroll bars get created precisely when they are needed, but as the window shrinks further, the scroll bar overcompensates and a large white space develops as you scroll to the bottom.
  8. UI2 Scroll bars in the Definition Editor get misaligned.  Some parts of large, long definitions are inaccessible.
  9. UI3 The tab heading for a new sequence does not update when you change the comment.
  10. UI3 The headings in the Dance Editor are misaligned.
  11. UI2 The text displayed for an ampersand is not correct (essentially the ampersands disappear).
  12. UI3 The Formation Editor will sometimes shift the displayed position of a formation if new spots are added (this sometimes shifts the spots completely out of view).  Saving and restarting the program fixes the problem.

Bugs Fixed

  1. SD2 Limited four-dancer stars are now supported.  Dancers in twin diamonds can be switched to four-dancer stars which can then be turned.  Twin four-dancer stars can subsequently be restored to diamonds.
  2. SD2 Two-ladies chain 3/4 now works.
  3. SD2 The Same Position Rule was not working correctly.  Calls that made dancers collide would fail with "Motions are not valid".
  4. SD2 From a quarter tag, calling "centers cast off 3/4' should yield a proper 2x4 arrangement.  It does now.
  5. UI1 The program could run out of fonts, but in practice this would require substantial amounts of usage (opening and closing lots of windows, printing lots of pages, etc.).
  6. UI1 Under some conditions when you have opened a call in the Call Analysis browser, the program will crash when you modify a definition (which should flush the analysis pages).

 

Alpha-2 Release Notes- June 3, 2010

This is the second Alpha release of Sets in Motion. 

Features Planned

  1. Some form of 'smart' editing of sequences.  Think of it as 'auto-complete' or 'intellisense' for calls.  This is a priority to develop.
  2. More designations.  This release reworked the parsing of ANYONE phrases so that 'beaus' and 'belles' now are marked as A-1 terms.  These changes also made it much easier to add new phrases.  For this release, for example, the following new phrases are now recognized:
    • lead ANYONE - designates dancers who are leaders as well as the following ANYONE phrase.  For example: 'lead girls', 'lead ends'.
    • end ANYONE - designates dancers who are ends in the current formation as well as the following ANYONE phrase.  For example: 'end boys', 'end heads'.
    • center ANYONE - designates dancers who are centers in the current formation as well as the following ANYONE phrase.  For example: 'center beaus'.
    Obviously, many more such phrases need to be added. 
  3. More calls.  The Plus level is up next.

Features Added

  1. The Mainstream list has been added to the definitions.
  2. A Formation Editor has been added.  Now when you expand the Formations section of a grammar and click on a formation name, a Formation Editor will be brought up in the right-hand sub-window.
  3. Formation diagrams have been added to the definitions.  Each valid formation pattern will include the formation to its right.
  4. There is now a button to delete a part of a definition variant.
  5. Buttons have been added to variants to move them up or down within the definition.
  6. Preferences have been added to give you control of the layout of a printed sequence.  You can set the various margins and fonts on the preferences page now and they will apply to the next printing (and will be retained until you change them again).

Known Bugs

  1. Four-dancer stars are not supported.
  2. Two ladies chain 3/4 cannot be expressed.
  3. Couples are not defined for infacing rings, when boy-girl arrangements should be able to do couple-based calls.
  4. From a squared set, the call 'turn your partner by the left 1/2' does not produce a thar.  It should.
  5. From a quarter tag, calling "centers cast off 3/4' does not yield a proper 2x4 arrangement.  It should.
  6. Dancers move through one another without collision.  This is also impossible. 

Bugs Fixed

  1. Scrollbars did not work properly if the section was too small.  Essentially clicking on the scrollbars would be interpreted as clicking on the enclosed window instead.
  2. Editing sequences could end up causing your typing to double, triple or even more, sssooo aaa ppphhhrrraaassseee lllooooookkkeeeddd llliiikkkeee ttthhhiiisss.
  3. Typos or otherwise malformed calls entered in the Sequence Editor could crash the program.  This was somewhat unpredictable.  Error recovery when parsing is now handled correctly.
  4. If you edited a long definition that required scrolling, the scroll bars would reset after each change.  They are preserved now.
  5. Column circulates did not work.  All dancers always stepped forward, even the leads of the columns.
  6. The call 'ends circulate' for parallel lines did not work.  It has been added.
  7. The call 'promenade home' from left-hand two-faced lines was not recognized.  It has been added.
  8. Fractionalizing some calls was not working.  This included things like 'split circulate 1 and 1/2'.
  9. You can now close and re-open dance files as many times as you like.
  10. You can now place empty lines in a sequence without interfering with the animation.
  11. Beaus and belles are now restricted to A-1 and above.
  12. A number of situations caused the generated motions of the dancers to be incorrect.  At least eight different bugs fell into this category and a detailed explanation of all of them is not worth the space.  At this point, all known issues with calls succeeding, but producing dis-continuous motion are fixed.

 

Alpha-1 Release Notes- May 16, 2010

This is the first public release of Sets in Motion.  Using a relatively standard terminology, this is an Alpha-quality release.  This means that some key features (like formation viewing/editing) are missing, and other functions (printer formatting) are incomplete.  The program has been tested, so most features that are documented are available and will likely work correctly.  Calls may have significant bugs in their definitions.  In particular, the starting formation may be incomplete or the existing definition language may not work correctly for some starting formations.

As soon as I deliver this version, I will begin working on the next, more mature release.  I plan on releasing several updates before calling this an officially released product.  Until then, expect to see occasional new updates.  I will be setting up a Web-site to make checking for and getting updates possible.  For now, this package is being distributed via e-mail to specific individuals.

Feedback Needed

I am interested in getting broad feedback on how the major components of the user interface flow together.  Does it make sense to you when shifting from editing a sequence to using the animation or call analyzer.  Are the labels on buttons clear?  Are there key features that you need that do not appear in the program.  I have my own ideas about the future direction of the program and have hinted at future features in places.  I do not want to bias you, however, so I have refrained from much prognostication.  I really want to know what you think would be useful (hopefully to most users).

Is the help text accurate and useful?  Do I need more detailed discussion of the program controls and displays?  Are the essays in the Caller's Guide helpful or confusing?  What topics would you have liked to see, but are missing?

Most importantly, try composing some sequences.  So far, the Basic list is pretty completely implemented.  I don't expect you to write much material but anything you could do to identify wrong or missing call variations will be very useful.

At this stage, the definitions and primitives are not thoroughly documented, so I would be amazed if you could figure out how to do more than minor tweeks.  The definition language is pretty unexpressive currently, so many more advanced concepts are simply impossible to express.

If you have any feedback or questions, e-mail me at bjervis@gmail.com.  Please check the following list of known issues before reporting anything.  When in doubt, report a problem.  In particular, none of the known issues cause the program to crash or lock up indefinitely.  The program should be responsive no matter what.  If you see any of these severe failures, please let me know about it.

The program should not corrupt your data.  Definitely let me know if you think you have saved something and it does not reload properly.  If your machine crashes while the program is writing data, I can't make any guarantees.  In general, the program will create a backup copy of any file before it tries to save a new copy, but there is no automated way to recover these backups.  If you think you have a corrupted file, do not save again.  That will destroy the backup.  Contact me if you need help finding and restoring the backup files.

Known Bugs

  • Four-dancer stars are not supported.
  • Two ladies chain 3/4 cannot be expressed.
  • Beaus and belles are not restricted to Advanced and above.
  • Closing a dance-file editor and re-opening it does not work correctly.  The file is locked out until you restart the program.  (You can close and re-open the My Library, Definitions and My Definitions files correctly.)
  • If you have a blank line in a sequence, the animation playback will 'get stuck' at the blank.  You will see the timer move forward and eventually stop, but the dancers freeze at the position of the blank line and then instantaneously jump to the final position when the animation completes.
